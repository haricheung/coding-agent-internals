# 模块 9：编程智能体的提示词工程

## 模块概述

> **核心问题：怎么跟编程代理"说话"才能让它高效工作？**

前面的模块拆解了 AI 编程代理的内部机制——它如何感知、规划、执行和反馈。但在实际使用中，**用户给代理的输入质量**往往比代理本身的架构更决定最终效果。一条精心设计的指令可以让代理一次成功，一条模糊的指令可以让代理在循环中浪费 10 万 token 后交出一团乱麻。

这就是编程智能体的提示词工程——不是通用的 "prompt engineering"，而是专门针对**代码生成代理**的指令设计方法论。

| 主题 | 核心问题 | 你将学到 |
|------|---------|---------|
| 系统提示词与指令文件 | 怎么给代理设定"人设"？ | 指令设计的认知模型、有效指令的六要素 |
| 高效的任务拆解 | 复杂任务怎么拆才对代理友好？ | 任务粒度的 U 型曲线、四条黄金法则 |
| 上下文管理 | 该给代理看什么、不看什么？ | 上下文的 ROI 分析、溢出时的四种策略 |
| 常见失效模式 | 代理为什么会"犯蠢"？ | 四种典型失效模式的识别与修正 |
| 高阶技巧 | 怎么精确控制代理行为？ | 约束生成、自我检查、多轮引导策略 |

## 9.1 系统提示词与指令文件

> 模块 7 已经介绍了 CLAUDE.md 的文件格式、三级继承体系和 settings.json 配置。本节聚焦**提示词工程**层面——如何设计让代理真正"听得懂、做得对"的指令。

### 指令设计的认知模型

给 AI 编程代理写指令，本质上是在做一件事：**把你脑中的隐性知识转化为代理可执行的显性规则**。

```
人类工程师的隐性知识:
  "这个项目的错误处理风格是返回 Result<T, E>，不用 throw"
  "我们的 API 路由都在 app/api/ 下面，用 Next.js App Router"
  "提交信息要用 conventional commits 格式"
  "测试用 vitest 跑，不用 jest"

        ↓ 提示词工程 ↓

代理可执行的显性规则:
  CLAUDE.md 中的明确指令
```

这个转化过程有三个层次：

```
层次 1: 规范层（What）
    → 告诉代理"做什么、不做什么"
    → 例："使用 TypeScript strict mode"、"不要使用 class 组件"

层次 2: 方法层（How）
    → 告诉代理"怎么做"
    → 例："运行测试用 pnpm test"、"数据库迁移用 prisma migrate dev"

层次 3: 判断层（Why & When）
    → 告诉代理"什么情况下做什么选择"
    → 例："如果修改了 API 路由，必须同时更新对应的 OpenAPI schema"
    → 例："性能关键路径中避免使用 ORM，直接写 SQL"
```

大多数人只写到层次 1（规范），但**层次 3（判断）才是让代理从"能用"到"好用"的关键**。

### 有效指令的六要素

一条好的代理指令应该满足以下六个要素：

```
┌──────────────────────────────────────────────────┐
│              有效指令的六要素                       │
├──────────────────────────────────────────────────┤
│                                                  │
│  1. 明确性（Specific）                             │
│     ✗ "代码要写好"                                 │
│     ✓ "函数不超过 30 行，圈复杂度不超过 10"          │
│                                                  │
│  2. 可执行性（Actionable）                         │
│     ✗ "注意代码质量"                               │
│     ✓ "每次 Edit 后运行 pnpm lint 检查"             │
│                                                  │
│  3. 可验证性（Verifiable）                         │
│     ✗ "确保没有 bug"                               │
│     ✓ "修改后运行 pnpm test 确保所有测试通过"        │
│                                                  │
│  4. 上下文感知（Context-aware）                    │
│     ✗ "使用主流的测试框架"                          │
│     ✓ "使用 vitest（已在 package.json 中配置）"     │
│                                                  │
│  5. 优先级清晰（Prioritized）                      │
│     ✗ 列出 50 条不分主次的规则                      │
│     ✓ "最重要：不要修改 core/ 目录下的文件"          │
│                                                  │
│  6. 异常处理（Exception-aware）                    │
│     ✗ "按规范写代码"                               │
│     ✓ "如果遇到类型错误无法解决，停下来问我"         │
│                                                  │
└──────────────────────────────────────────────────┘
```

### 对比：好指令 vs 坏指令

**坏的 CLAUDE.md（模糊、不可执行）：**

```markdown
# CLAUDE.md
这是一个 Web 项目。请写高质量的代码，注意安全性和性能。
使用 React 和 TypeScript。
```

**好的 CLAUDE.md（明确、可执行、有判断指导）：**

```markdown
# CLAUDE.md

## 项目概述
Next.js 14 电商平台，TypeScript + Prisma + PostgreSQL。
当前阶段：MVP 开发中，优先功能完整性而非性能优化。

## 代码规范
- TypeScript strict mode（tsconfig.json 已配置）
- React 函数式组件 + hooks，禁止 class 组件
- 状态管理：简单状态用 useState，跨组件状态用 Zustand（已安装）
- 样式：Tailwind CSS，不要用 inline style 或 CSS modules

## 常用命令
- pnpm dev — 启动开发服务器（端口 3000）
- pnpm test — 运行单元测试（vitest）
- pnpm lint — ESLint + Prettier 检查
- pnpm db:push — 同步 Prisma schema 到数据库

## 关键约定
- API 路由在 app/api/ 下，使用 Route Handlers（不是 Pages API）
- 所有 API 输入用 Zod 做运行时校验
- 数据库查询统一在 lib/db/ 中封装，组件不直接调用 Prisma
- 错误处理：API 返回 { error: string, code: string }，不要抛异常

## 修改须知
- 修改 prisma/schema.prisma 后必须运行 pnpm db:push
- 修改 API 路由后必须更新 lib/api/client.ts 中对应的类型定义
- 不要修改 lib/core/ 下的文件（核心支付逻辑，需要人工审查）
```

关键差异：好指令告诉代理**做了某件事之后还需要做什么**（修改 schema → 运行 db:push），以及**什么不能做**（不修改 lib/core/）。这些"连锁规则"和"禁区"是最容易被遗漏但最能避免事故的指令。

### CLAUDE.md vs .opencode 配置对比

两种工具的指令系统设计哲学有本质差异：

| 维度 | CLAUDE.md | .opencode 配置 |
|------|-----------|---------------|
| 格式 | 自由文本 Markdown | 结构化 JSON/YAML |
| 表达力 | 可以写任意自然语言规则 | 受限于预定义字段 |
| 粒度 | 项目级 + 目录级分层 | 项目级 |
| 生效方式 | 注入到 System Prompt | 解析为工具配置 |
| 适合内容 | 工程判断、架构约定、工作流规则 | 模型选择、工具配置、Provider 设置 |

```
CLAUDE.md 的优势：自然语言的灵活性
────────────────────────────────
"如果你在修复一个 bug，先写一个能复现该 bug 的测试，
确认测试失败，然后再修复代码，最后确认测试通过。"

→ 这种多步骤的条件性工作流，在结构化配置中很难表达。

.opencode 的优势：精确的工具控制
────────────────────────────────
{
  "models": {
    "fast": { "provider": "anthropic", "model": "claude-haiku-4-5" },
    "smart": { "provider": "anthropic", "model": "claude-opus-4-6" }
  }
}

→ 模型切换、Provider 配置这类参数化设定，结构化格式更合适。
```

**核心洞察**：CLAUDE.md 本质是一个**嵌入到对话上下文中的 System Prompt 扩展**。它的设计原则与通用 prompt engineering 相同——清晰、具体、有优先级。但它多了一个独特优势：**它可以随代码库一起版本管理**，团队可以像 review 代码一样 review 指令。

### 指令覆盖规则

当多级指令冲突时，需要理解覆盖机制：

```
优先级（从高到低）：

1. 用户当前对话中的显式指令
   > "这次不要跑测试，直接改"
   → 覆盖 CLAUDE.md 中"每次修改后跑测试"的规则

2. 目录级 CLAUDE.md
   src/legacy/CLAUDE.md: "这个目录用 JavaScript，不用 TypeScript"
   → 覆盖项目级的 "使用 TypeScript" 规则

3. 项目级 CLAUDE.md
   → 项目的默认规则

4. 用户级 ~/.claude/CLAUDE.md
   → 跨项目的个人偏好（如"提交信息用中文"）

5. 系统默认行为
   → Claude Code 内置的工具使用策略
```

**设计建议**：把**稳定的、跨文件的规则**放在项目级 CLAUDE.md，把**特定模块的例外规则**放在目录级 CLAUDE.md。避免在对话中反复说明应该固化到文件中的规则。

## 9.2 高效的任务拆解

### 任务粒度的 U 型曲线

给代理的任务存在一个最优粒度——太粗和太细都会降低成功率：

```
成功率
  │
  │        ★ 最优粒度
  │       ╱  ╲
  │      ╱    ╲
  │     ╱      ╲
  │    ╱        ╲
  │   ╱          ╲
  │  ╱            ╲
  │ ╱              ╲
  │╱                ╲
  └──────────────────────→ 任务粒度
  太细              太粗

太细的问题：                    太粗的问题：
- 上下文频繁切换                 - 超出上下文窗口
- 全局一致性丧失                 - 规划失败/遗漏
- 人工协调成本高                 - 中途迷失方向
- 例："把第 42 行的                - 例："重构整个认证系统，
  变量名改成 x"                    包括 SSO、2FA、权限"
```

### 四条黄金法则

**法则 1：一个任务 = 一个清晰的验收条件**

```
❌ "改进用户模块"
   → 什么叫"改进"？没有验收标准，代理不知道什么时候停下来。

✓ "给 POST /api/users 添加邮箱格式校验。
    校验失败返回 400 和错误信息。
    添加对应的单元测试。"
   → 验收条件明确：API 行为 + 测试覆盖。
```

**法则 2：提供"锚点"而非"地图"**

不要试图给代理规划每一步（代理自己会规划），而是提供关键的定位信息：

```
❌（过度规划——把代理当 copilot）:
   "1. 打开 src/auth/login.ts
    2. 找到第 45 行的 validatePassword 函数
    3. 在 if 语句前添加一行 null check
    4. 保存文件
    5. 运行 npm test"

✓（提供锚点——让代理自己导航）:
   "修复 login 接口的 500 错误。
    错误日志：TypeError: Cannot read property 'email' of null
    出现在 src/auth/login.ts 的 validatePassword 中。
    修复后确保 npm test 通过。"
```

关键区别：好的提示给出**问题描述**（是什么）+ **定位线索**（在哪里）+ **验收标准**（怎么算完成），但不规定具体操作步骤。让代理自己 Grounding → Planning → Action。

**法则 3：边界清晰化——明确"不要做什么"**

代理倾向于"多做"——给它修一个 bug，它可能顺手重构周围的代码。明确边界可以防止这种行为：

```
✓ "修复 validatePassword 的 null 引用错误。
    只修改 src/auth/login.ts 和对应的测试文件。
    不要重构现有的错误处理模式。
    不要添加新的依赖。"
```

**法则 4：复杂任务拆成"对话序列"而非"一次性指令"**

对于大型任务，与其写一个超长的提示词，不如按阶段推进：

```
对话 1（Grounding 阶段）:
> "分析 src/auth/ 目录下的认证模块，告诉我当前的架构、
>  主要的数据流、和你认为的潜在问题。不要修改任何文件。"

    ← 代理输出架构分析

对话 2（Planning 阶段）:
> "基于你的分析，我想添加 OAuth2 Google 登录。
>  给我一个实现计划，列出需要修改的文件和步骤。
>  先不要动手。"

    ← 代理输出实现计划，用户审查/修正

对话 3（Action 阶段）:
> "按照计划实现第 1-3 步。完成后运行测试确认。"

    ← 代理执行，用户验证

对话 4（Action 续）:
> "第 1-3 步看起来不错。继续实现第 4-6 步。"
```

这种"分阶段对话"的优势：
- 每一轮的上下文窗口都用在当前阶段
- 用户可以在每个检查点纠偏
- 避免代理在超长任务中迷失

### 任务拆解模板

对于中等复杂度的功能开发，推荐以下模板：

```markdown
## 任务描述
[一句话说明要做什么]

## 背景信息
- 相关文件：[列出 2-5 个关键文件路径]
- 现有行为：[当前是什么样的]
- 期望行为：[改完应该是什么样的]

## 约束条件
- 只修改以下目录：[...]
- 不要修改：[...]
- 依赖限制：[不要添加新依赖 / 只用已安装的 xxx]

## 验收标准
1. [具体的功能验证条件]
2. [测试要求]
3. [其他质量要求]
```

### 对代理友好 vs 不友好的任务特征

| 特征 | 对代理友好 | 对代理不友好 |
|------|-----------|-------------|
| 范围 | 明确的文件/模块边界 | "整个项目" |
| 验收 | 可测试的具体行为 | "看起来好一点" |
| 参考 | 有类似代码可模仿 | 完全从零创造 |
| 依赖 | 自包含，少外部依赖 | 需要协调多个系统 |
| 反馈 | 有测试/lint 可验证 | 需要人类主观判断 |
| 上下文 | 所需信息在代码库中 | 需要外部知识（设计稿、会议纪要） |

## 9.3 上下文管理

### 上下文窗口的注意力模型

LLM 的上下文窗口不是一个"平等对待每个 token"的容器。实际的注意力分布更接近：

```
注意力强度
  │
  │███                                              ███
  │███                                              ███
  │███                                              ███
  │███                                              ███
  │████                                            ████
  │██████                                        ██████
  │████████▅▅▅▃▃▃▃▃▃▃▃▃▃▃▃▃▃▃▃▃▃▃▃▃▃▃▃▃▃▃▃▃▅▅████████
  └────────────────────────────────────────────────────→
  窗口开头          窗口中间                    窗口末尾
  (System Prompt)   ("Lost in the Middle")     (最近消息)
```

这就是 "Lost in the Middle" 效应——窗口中间的信息最容易被忽略。这对上下文管理有直接影响：

**设计原则**：
- **最重要的指令放在 System Prompt（窗口开头）** — CLAUDE.md 的内容
- **当前任务的关键信息放在最近的消息中（窗口末尾）**
- **大段参考代码放在中间** — 它们会被"部分关注"，但细节可能丢失

### 上下文内容的 ROI 分析

不是所有信息都值得放入上下文。每类信息都有其 token 成本和信息收益：

```
                    高收益
                      │
         ┌────────────┼────────────┐
         │  ★ 甜蜜区  │            │
         │            │            │
         │ ·错误日志   │ ·相关源码   │
         │ ·类型定义   │ ·测试文件   │
         │ ·API 契约   │            │
低成本 ──┼────────────┼────────────┤── 高成本
         │            │            │
         │ ·项目结构   │ ·整个文件   │
         │ ·命令列表   │ ·生成代码   │
         │            │ ·node_modules│
         │            │            │
         └────────────┼────────────┘
                      │
                    低收益
```

**甜蜜区（高收益 + 低成本）**：

| 内容类型 | 典型大小 | 收益 |
|---------|---------|------|
| 错误日志/Stack trace | 20-50 行 | 直接定位问题 |
| 接口/类型定义 | 50-100 行 | 理解 API 契约 |
| 相关函数签名 | 10-30 行 | 保持调用一致性 |
| 目录结构概览 | 20-40 行 | 导航全局 |

**陷阱区（低收益 + 高成本）**：

| 内容类型 | 典型大小 | 问题 |
|---------|---------|------|
| 完整大文件 | 500+ 行 | 大量无关代码稀释注意力 |
| 生成的代码 | 1000+ 行 | 噪声极高（lock 文件、.d.ts） |
| 冗长的讨论历史 | 变长 | 过时信息干扰判断 |

### 上下文溢出的信号

当对话过长时，代理会出现以下症状：

```
溢出信号                          表现
──────────                      ──────
1. 遗忘早期指令        代理开始违反 CLAUDE.md 中的规则
2. 重复操作            读取已经读过的文件
3. 前后矛盾            先说"方案 A 更好"，后来又选了方案 B
4. 忽略约束            你说了"只改这个文件"，它改了其他文件
5. 质量下降            代码风格开始不一致，变量命名变差
```

### 四种上下文管理策略

**策略 1：主动压缩（/compact）**

当对话变长时，主动使用 `/compact` 命令压缩历史：

```
使用时机:
- 对话超过 20 轮
- 代理开始出现溢出信号
- 准备开始一个新子任务

效果:
- 将之前的对话压缩为摘要
- 保留关键决策和上下文
- 释放窗口空间给新任务

注意:
- 压缩会丢失细节——压缩前确认重要信息已被执行
- 可以在压缩前告诉代理"请记住以下关键决策：..."
```

**策略 2：新会话切割**

对于多阶段大任务，在自然边界处开启新会话：

```
Session 1: 分析现状 + 制定方案
    → 输出：实现计划（文字记录）

Session 2: 实现阶段 1-3
    → 输入：实现计划 + 相关代码路径
    → 输出：已完成的代码变更

Session 3: 实现阶段 4-6 + 测试
    → 输入：计划中剩余步骤 + 已变更文件列表
    → 输出：完整功能 + 测试

关键: 在新 Session 开头提供简洁的上下文摘要，
而不是让代理"重新发现"所有信息。
```

**策略 3：分层注入**

按需提供信息，而不是一次性灌入所有上下文：

```
第一轮: 只提供问题描述和关键文件路径
    → 让代理自己 Grounding（Glob + Grep + Read）
    → 代理会基于需要选择性地读取信息

第二轮: 代理问"我需要了解 X 的实现"
    → 你提供针对性的上下文补充

优势: 代理自己搜索到的信息，比你直接灌入的信息，
在注意力分配上更有优势——因为搜索过程本身建立了上下文关联。
```

**策略 4：上下文标记**

在长对话中，用显式标记帮助代理定位关键信息：

```
> ⚠️ 关键约束（请在整个任务中遵守）:
> 1. 不要修改 core/ 目录
> 2. 所有新接口必须有 Zod schema
> 3. 测试必须通过后才能提交

> 📋 当前任务:
> 实现用户头像上传功能

> 📁 相关文件:
> - app/api/user/route.ts（现有用户 API）
> - lib/storage/s3.ts（S3 上传工具，已实现）
> - prisma/schema.prisma（需要添加 avatarUrl 字段）
```

使用 emoji 标记或 Markdown 标题来**结构化**你的输入，帮助模型的注意力机制快速定位不同类型的信息。

### 上下文预算的心智模型

把上下文窗口想象成一个固定预算：

```
200K token 窗口的典型分配：

┌─────────────────────────────────────┐
│ System Prompt + CLAUDE.md   ~5K    │ ← 固定开销
├─────────────────────────────────────┤
│ 代理读取的代码文件        ~50-100K  │ ← Grounding 消耗
├─────────────────────────────────────┤
│ 对话历史                  ~30-50K   │ ← 随轮次增长
├─────────────────────────────────────┤
│ 代理的思考过程（extended   ~20-30K  │ ← 模型内部
│   thinking）                        │
├─────────────────────────────────────┤
│ 剩余可用空间              ~20-80K   │ ← 越多越好
└─────────────────────────────────────┘

当"剩余可用空间"接近 0 → 触发自动压缩或性能下降
```

**实用建议**：用 `/cost` 命令监控当前会话的 token 消耗。当总消耗超过 100K token 时，考虑压缩或切换新会话。

## 9.4 常见失效模式

### 失效模式 1：API 幻觉

**症状**：代理调用了不存在的函数、使用了错误的参数签名、引用了虚构的库。

```python
# 代理生成的代码（错误）
from fastapi.security import OAuth2AuthorizationCodeBearer

# 实际上 FastAPI 没有这个类
# 正确的是 OAuth2AuthorizationCodeBearer 确实存在，
# 但代理可能把参数搞错：
bearer = OAuth2AuthorizationCodeBearer(
    authorizationUrl="...",   # ✓ 存在
    tokenUrl="...",           # ✓ 存在
    refreshUrl="...",         # ✗ 不存在这个参数（实际叫 refresh_url？）
    auto_error=True           # ✓ 存在
)
```

**根因分析**：

```
API 幻觉的来源:
├── 训练数据过时 — 模型学习的是旧版本 API
├── 模式混淆 — 将 A 库的 API 模式"嫁接"到 B 库
├── 参数推测 — 基于命名惯例猜测参数名
└── 版本混淆 — 混合了 v1 和 v2 的 API
```

**修正策略**：

```markdown
在 CLAUDE.md 中指定版本和关键 API:

## 依赖版本（请严格遵守，不要使用其他版本的 API）
- Next.js 14.1（App Router，非 Pages Router）
- Prisma 5.x（使用 prisma.model.findUnique，不是 findOne）
- Zod 3.x

## 常见陷阱
- NextResponse.json() 不接受 status 参数，用第二个参数传 { status: 400 }
- Prisma 5 中 rejectOnNotFound 已移除，用 findUniqueOrThrow 替代
```

也可以在任务提示中直接引入正确的 API 参考：

```
> 实现文件上传功能。参考 S3 SDK 用法：
> https://docs.aws.amazon.com/sdk-for-javascript/v3/...
> 使用 @aws-sdk/client-s3 v3（不是 aws-sdk v2）
```

### 失效模式 2：过度工程

**症状**：代理生成了远超需求的代码——你要一个简单函数，它给你一个完整的抽象层。

```
你的请求:
> "写一个函数，把用户名转成 slug（小写 + 连字符）"

代理的过度工程输出:
├── SlugGenerator 类（支持多种策略）
├── SlugStrategy 接口
├── DefaultSlugStrategy
├── ChineseSlugStrategy（支持中文拼音转换）
├── SlugCache（缓存已生成的 slug）
├── SlugValidator（验证 slug 唯一性）
├── slug.test.ts（15 个测试用例）
└── slug.config.ts（配置文件）

你实际需要的:
function toSlug(name: string): string {
  return name.toLowerCase().replace(/[^a-z0-9]+/g, '-').replace(/^-|-$/g, '');
}
```

**根因分析**：

LLM 在训练中见过大量"完善的"、"生产级"的代码，倾向于生成看起来"专业"的过度抽象。此外，模型缺乏对**当前项目阶段和实际需求规模**的判断。

**修正策略**：

```
在提示词中明确"简单优先"原则:

> "用最简单的方式实现。不要创建额外的类、接口或配置文件。
>  如果能用一个函数解决，就不要用一个类。
>  不要添加当前不需要的功能。"

或者更精确:

> "这个功能只在一个地方使用。写一个函数就够了，
>  不需要抽象层、配置系统或缓存。"
```

**CLAUDE.md 级别的防御**：

```markdown
## 工程原则
- YAGNI: 不要实现当前不需要的功能
- 三次法则：相同逻辑出现 3 次以上才考虑抽象
- MVP 阶段：优先功能完整性，不做过早优化
```

### 失效模式 3：死循环

**症状**：代理在修复一个问题时引入新问题，然后修复新问题时又引回原问题，循环往复。

```
典型的死循环模式:

Round 1: 修改代码以修复 Bug A
Round 2: 运行测试 → 测试 B 失败了
Round 3: 修改代码以修复测试 B → Bug A 又回来了
Round 4: 运行测试 → 测试 A 失败了
Round 5: 修改代码以修复测试 A → 测试 B 又失败了
... （无限循环）

Token 消耗: 每轮 ~2000 tokens
10 轮后: ~20,000 tokens 浪费，问题未解决
```

**根因分析**：

```
死循环的常见诱因:
├── 相互耦合的约束 — 满足 A 就破坏 B
├── 信息丢失 — 每轮修复时忘记之前为什么改
├── 策略固着 — 反复尝试同一类修复方案
└── 缺乏全局视角 — 只看到局部失败，不理解系统性问题
```

**修正策略**：

```
1. 在 CLAUDE.md 中设置"死循环断路器":

## 工作规则
如果同一个测试反复失败超过 3 次，停下来分析根因，
不要继续盲目修改。向我报告：
- 你尝试了什么
- 为什么没有效果
- 你认为根本原因是什么

2. 在对话中识别到循环时立即干预:

> "你已经在这个问题上循环了 3 轮。暂停修改。
>  请先分析为什么 A 和 B 互相冲突，
>  找到一个能同时满足两者的方案。"

3. 引导代理换思路:

> "之前的修改方式不行。换一个完全不同的方案：
>  不要修改 validateInput，而是在调用它之前
>  做数据预处理。"
```

### 失效模式 4：Token 浪费

**症状**：代理消耗了大量 token 却没有实质进展——反复读取相同的文件、生成冗长的解释、执行不必要的搜索。

```
Token 浪费的典型模式:

模式 A: 重复读取
  Read(file_a.ts)     ← 第一次
  Read(file_b.ts)
  Read(file_a.ts)     ← 第二次（完全相同的文件）
  Read(file_c.ts)
  Read(file_a.ts)     ← 第三次...

模式 B: 过度搜索
  Glob("**/*.ts")     ← 搜出 500 个文件
  Grep("function")    ← 搜出 2000 个匹配
  → 代理："我找到了很多相关文件，让我继续搜索..."
  Grep("export")      ← 又搜出 1500 个匹配

模式 C: 冗长输出
  "让我详细解释一下我发现了什么。
   首先，我注意到 file_a.ts 中有一个函数叫 handleRequest，
   这个函数在第 45 行定义，它接受一个 Request 参数，
   然后在第 47 行调用了 validateInput，
   validateInput 在 file_b.ts 的第 12 行定义，
   ..."
  → 500+ tokens 的"分析"，只为了说"handleRequest 调用了 validateInput"
```

**修正策略**：

```
1. 精确的搜索指令:

❌ "找到认证相关的代码"
✓ "在 src/auth/ 目录下找到 validateToken 函数的定义"

2. 限制输出冗长度:

> "直接修改代码，不需要解释你发现了什么。
>  如果有重要的设计决策需要沟通，用一句话概括。"

3. 提供定向入口:

> "相关代码在 src/auth/middleware.ts 第 30-60 行。
>  直接阅读该文件，不需要搜索。"
```

### 失效模式总结表

| 失效模式 | 识别信号 | 根因 | 修正策略 |
|---------|---------|------|---------|
| API 幻觉 | 编译错误、运行时 ImportError | 训练数据过时/模式混淆 | 指定版本、提供 API 参考 |
| 过度工程 | 代码量远超预期、不必要的抽象 | 模型偏好"专业"代码 | 明确简单优先、YAGNI 原则 |
| 死循环 | 同一测试反复失败、改了又改 | 耦合约束、策略固着 | 设置断路器、引导换思路 |
| Token 浪费 | 高消耗低进展、重复操作 | 搜索过泛、输出冗长 | 精确指令、提供入口 |

## 9.5 高阶技巧

### 技巧 1：约束生成

约束生成（Constrained Generation）是指通过显式约束来限制代理的行动范围。这是最直接有效的控制手段。

**文件范围约束**：

```
> "只修改以下文件，不要创建新文件，不要修改其他文件：
>  - src/api/users.ts
>  - src/api/users.test.ts"
```

**技术约束**：

```
> "使用 fetch API，不要安装 axios。
>  使用原生 crypto 模块，不要安装 bcrypt。
>  错误处理用 Result<T, E> 模式，不要 try-catch。"
```

**行为约束**：

```
> "先写测试，再写实现。先让测试失败，确认测试逻辑正确后再实现功能。"

> "不要运行 git commit。改完代码后停下来，让我检查。"

> "不要使用 Bash 工具执行任何破坏性操作（rm、git reset 等）。"
```

**输出格式约束**：

```
> "修改完成后，列出所有改动的文件和每个文件的改动摘要，
>  格式如下：
>  - [文件路径]: [一句话描述改动]"
```

### 技巧 2：让代理自我检查

在执行任务后，要求代理自我审查可以捕获很多错误：

```
> "实现完成后，请自己检查以下几点：
>  1. 所有新增的函数是否有错误处理？
>  2. 是否有硬编码的值应该提取为常量？
>  3. 类型定义是否完整（没有 any）？
>  4. 边界条件是否处理了（空数组、null、超长字符串）？
>  如果发现问题，直接修复。"
```

更高级的自检模式——**红队审查**：

```
> "现在假设你是一个代码审查者（不是作者）。
>  审查你刚才写的代码，找出至少 3 个可以改进的地方。
>  关注：安全漏洞、性能问题、边界条件、可读性。
>  然后修复你发现的问题。"
```

这利用了 LLM 的一个特性：**换角色后，模型会用不同的视角重新审视同一段代码**，经常能发现"作者视角"时忽略的问题。

### 技巧 3：多轮对话中的引导策略

在多轮对话中，如何有效引导代理而不丢失方向：

**渐进式细化**：

```
轮次 1（宽泛）:
> "我想给这个 Express 应用添加速率限制"

轮次 2（了解方案后收窄）:
> "方案看起来不错，但不要用 Redis，用内存存储就行。
>  这是一个单实例应用。"

轮次 3（微调细节）:
> "速率限制改成每个 IP 每分钟 30 次请求。
>  超限时返回 429 和 Retry-After 头。"
```

**方向纠正**：

当代理走偏时，不要只说"不对"，要给出**正确方向**：

```
❌ "这不是我想要的，重做。"
   → 代理不知道哪里不对，可能犯同样的错

✓ "你用了继承来实现多态，但这个项目的惯例是用组合。
    请改用策略模式——定义一个 Strategy 接口，
    让不同实现作为独立的函数传入。
    参考 src/services/payment.ts 中的已有模式。"
```

**检查点确认**：

```
> "在修改数据库 schema 之前，先告诉我你打算怎么改，
>  等我确认后再执行 prisma migrate。"
```

### 技巧 4：Plan Mode 工作流

Claude Code 的 Plan Mode（通过 `/plan` 或 Shift+Tab 触发）是一种强制代理"先想后做"的机制：

```
Plan Mode 工作流:

1. 进入 Plan Mode
   → 代理只能使用只读工具（Read/Glob/Grep/WebSearch）
   → 不能修改文件、不能执行命令

2. 代理输出实现计划
   → 列出需要修改的文件
   → 描述每个修改的内容
   → 预估影响范围

3. 用户审批/修正
   → 确认计划合理
   → 调整优先级或方案

4. 退出 Plan Mode，代理按计划执行

适用场景:
- 不熟悉的代码库中的复杂修改
- 涉及数据库 schema 变更的功能
- 多文件的架构重构
- 任何你不确定代理会怎么做的场景
```

### 技巧 5：错误信息的正确传递

当代理遇到错误时，如何给出有效的错误信息：

```
❌ 低效的错误传递:
> "报错了，请修复"

✓ 高效的错误传递:
> "运行 pnpm test 后，test/auth.test.ts 的第 3 个测试失败：
>
>  FAIL  test/auth.test.ts
>  ● Auth › should reject expired tokens
>    Expected: 401
>    Received: 200
>
>  这说明 token 过期检查逻辑有问题。
>  检查 src/auth/jwt.ts 中的 verifyToken 函数。"
```

**关键原则**：给代理**完整的错误输出** + **你的初步判断**。代理擅长从错误信息中定位问题，但它需要你提供运行时的反馈数据。

### 技巧 6：跨 Session 的知识传递

当一个任务跨多个会话完成时，如何有效传递知识：

```
方式 1: CLAUDE.md 作为"笔记本"
──────────────────────────────
在 CLAUDE.md 中添加一个"当前进度"区块：

## 当前进度（完成后删除此节）
- [x] 用户认证 API（已实现，在 src/api/auth/）
- [x] JWT 中间件（已实现，在 src/middleware/auth.ts）
- [ ] 刷新 token 逻辑（下一步）
- [ ] 前端登录页面

### 已知问题
- token 过期时间目前硬编码为 1h，需要改为可配置

方式 2: 新 Session 开头的上下文注入
──────────────────────────────────
> "我在上一个 session 中实现了用户认证功能。
>  相关文件：src/api/auth/、src/middleware/auth.ts
>  接下来请实现刷新 token 逻辑：
>  - 在 POST /api/auth/refresh 中接受 refresh token
>  - 验证后签发新的 access token
>  - refresh token 有效期 7 天
>  已知约束：参考现有的 JWT 工具函数 src/lib/jwt.ts"
```

## 本模块小结

### 核心要点回顾

```
提示词工程的三个层次
├── 规范层 (What) — 做什么、不做什么
├── 方法层 (How) — 怎么做
└── 判断层 (Why & When) — 什么情况下做什么选择

有效指令六要素
├── 明确性 — 具体的数值和行为，不用"好"、"高质量"
├── 可执行性 — 代理能采取的具体动作
├── 可验证性 — 有客观的通过/失败标准
├── 上下文感知 — 基于项目实际情况，不用"主流"
├── 优先级清晰 — 最重要的规则突出标注
└── 异常处理 — 遇到无法决策的情况时怎么办

任务拆解四法则
├── 一个任务 = 一个验收条件
├── 提供锚点而非地图
├── 边界清晰化（明确"不要做什么"）
└── 复杂任务拆成对话序列

上下文管理策略
├── /compact 主动压缩
├── 新 Session 切割（在自然边界处）
├── 分层注入（按需提供信息）
```
