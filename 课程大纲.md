# 课程设计计划：高级 AI 编程代理 — 机制、策略与实现

## Context

用户是工程与算法双修的技术专家，其领导要求准备关于 Claude Code、OpenCode 及其底层模型的高级课程。通过前期讨论，我们已经提炼出核心框架：

1. **核心循环**：Intent → Grounding（感知现状）→ Planning → Action → Grounding（观察结果）→ Feedback
2. **每个环节都有策略空间**，抓住机制+关键策略就抓住全部
3. **AlphaGo/MCTS 类比**可作为理论升华的锚点
4. 课程需要同时体现**工程深度**（怎么实现的）和**算法深度**（为什么这么实现）

## 课程总纲

### 课程名称：掀起 AI 编程的"引擎盖" — 机制、策略与工程实现

### 课程定位
- **受众**：有编程经验的工程师、算法工程师、技术管理者
- **前置要求**：有 LLM 使用经验，了解基本的 prompt engineering
- **目标**：学员能理解 AI 编程代理的完整机制，能评估/选择/定制工具，能基于原理构建自己的方案

### 课程结构：3 大篇章 + 9 个模块

---

## 第一篇：原理篇 — 引擎是怎么转的（模块 0-2）

### 模块 0：核心机制 — AI 编程代理的"发动机"

**工程视角**：一个 AI 编程代理到底在做什么？拆解 Claude Code 处理一个真实任务的完整 trace。

**算法视角**：序贯决策框架。

| 主题 | 内容 |
|------|------|
| 0.1 从一个真实任务说起 | 录屏/trace 回放：给 Claude Code 一个 bug fix 任务，逐帧拆解它做了什么 |
| 0.2 核心循环 | Intent → Grounding → Planning → Action → Grounding → Feedback 的完整推导 |
| 0.3 MDP 建模 | 将编程代理建模为马尔可夫决策过程 (S, A, T, R)，状态=代码库快照，动作=工具调用，奖励=测试结果 |
| 0.4 策略空间总览 | 每个环节的策略菜单（一张全景表），后续模块逐一展开 |
| 0.5 与 AlphaGo 的结构类比 | 策略网络≈LLM，价值网络≈测试/评估，MCTS≈搜索策略；类比成立的部分与裂缝 |

**关键论文**：ReAct (Yao 2022), Chain-of-Thought (Wei 2022)

**实操**：学员动手 trace 一个 Claude Code session，标注每一步属于循环的哪个环节

---

### 模块 1：Grounding — 感知的策略空间

**核心问题**：怎么让模型"看到"代码库？看什么、看多少、怎么看？

| 主题 | 内容 |
|------|------|
| 1.1 为什么 Grounding 是瓶颈 | SWE-agent 的核心洞察：Agent-Computer Interface 比模型更重要 |
| 1.2 策略 1：暴力检索 | Glob/Grep/全文件读取 — Claude Code 的方案，简单有效但消耗 token |
| 1.3 策略 2：结构化感知 | AST 解析、LSP 语义信息 — OpenCode 的方案，精准但实现复杂 |
| 1.4 策略 3：向量检索（RAG） | Embedding + 向量数据库，适合超大代码库，但有语义漂移风险 |
| 1.5 策略 4：混合方案 | 先粗检索再精读，分层感知策略 |
| 1.6 上下文窗口管理 | 有限窗口下的信息压缩、裁剪、优先级排序 |

**关键论文**：SWE-agent (Yang 2024), Agentless (Xia 2024)

**工程实操**：
- 对比 Claude Code (Grep+Read) 与 OpenCode (LSP) 在同一任务上的 grounding 差异
- 测量不同策略的 token 消耗和信息覆盖率

---

### 模块 2：底层模型 — 引擎的"燃料"

**核心问题**：模型的哪些能力在支撑代理行为？

| 主题 | 内容 |
|------|------|
| 2.1 代码预训练 | 从 Codex 到 Claude：在代码上预训练为什么有效 |
| 2.2 指令遵循与 RLHF | 模型怎么学会"听话"的 — RLHF 与 Constitutional AI |
| 2.3 Tool Use 能力 | 函数调用的训练机制：从自然语言到结构化 JSON 输出 |
| 2.4 长上下文推理 | 128K/200K 窗口的技术实现与实际表现衰减 |
| 2.5 模型选择策略 | Opus vs Sonnet vs Haiku：能力/成本/延迟三角；OpenCode 的多模型切换 |
| 2.6 基准测试的真相 | SWE-bench、HumanEval 在测什么、没测什么、怎么被刷榜 |

**关键论文**：Codex (Chen 2021), RLHF (Ouyang 2022), Constitutional AI (Bai 2022), Toolformer (Schick 2023)

**实操**：同一个编程任务分别用 Opus、Sonnet、Haiku 完成，对比行为差异和成本

## 第二篇：策略篇 — 每个环节怎么做（模块 3-6）

### 模块 3：Planning — 规划的策略空间

**核心问题**：知道现状和目标后，怎么拆解路径？

| 主题 | 内容 |
|------|------|
| 3.1 扁平任务列表 | 最简方案：直接列出 TODO，线性执行 |
| 3.2 树状分解 | Tree of Thoughts：展开多个方案分支，评估后选最优 |
| 3.3 DAG 依赖图 | 任务间有依赖关系时的拓扑排序执行 |
| 3.4 多代理分工 | MetaGPT 式角色分工 vs AutoGen 式对话协作 |
| 3.5 Plan 的粒度 | 什么时候细规划、什么时候粗规划？过度规划的代价 |
| 3.6 动态重规划 | 执行过程中发现计划不对，怎么调整？ |

**关键论文**：Tree of Thoughts (Yao 2023), MetaGPT (Hong 2023), AutoGen (Wu 2023)

**实操**：Claude Code 的 Plan Mode 实战 — 让代理先规划、审批后执行

---

### 模块 4：Action — 执行的策略空间

**核心问题**：决定了做什么之后，怎么做？

| 主题 | 内容 |
|------|------|
| 4.1 策略 1：Tool Use (JSON 函数调用) | Claude Code 的核心方案：Read/Edit/Write/Bash 工具集的设计哲学 |
| 4.2 策略 2：CodeAct (代码执行) | 用 Python/Shell 代码作为动作语言，比 JSON 更灵活 |
| 4.3 策略 3：直接生成 | 不调用工具，直接输出完整代码块，简单但缺乏验证 |
| 4.4 工具设计原则 | 什么样的工具接口对模型友好？从 SWE-agent 的 ACI 设计学到的 |
| 4.5 MCP 协议 | Claude Code 的 MCP Server 扩展机制：怎么给代理"装新工具" |
| 4.6 权限与沙箱 | 行动的安全边界：Claude Code 的权限模型 vs OpenCode 的设计 |

**关键论文**：CodeAct (Wang 2024), SWE-agent (Yang 2024)

**实操**：
- 构建一个自定义 MCP Server（如数据库查询工具）并接入 Claude Code
- 对比 Tool Use 和 CodeAct 两种策略在同一任务上的表现

---

### 模块 5：Feedback — 评估与修正的策略空间

**核心问题**：怎么知道做对了？做错了怎么办？

| 主题 | 内容 |
|------|------|
| 5.1 策略 1：外部验证 | 测试执行、Lint 检查、类型检查 — 最可靠但覆盖面有限 |
| 5.2 策略 2：模型自评 | LLM 审查自己的输出 — Self-Refine 的机制 |
| 5.3 策略 3：Reflexion | 从失败中学习：将错误经验转化为语言记忆，下次避免 |
| 5.4 策略 4：人在回路 | Human-in-the-loop：什么时候该问人？Claude Code 的权限确认设计 |
| 5.5 Hooks 系统 | Claude Code 的 pre/post hooks：在工具调用前后插入自动检查 |
| 5.6 失败恢复策略 | 重试 vs 回溯 vs 重新规划 vs 上报人类 — 怎么选？ |

**关键论文**：Self-Refine (Madaan 2023), Reflexion (Shinn 2023)

**实操**：配置 Claude Code 的 Hooks，实现"每次编辑后自动运行测试"

---

### 模块 6：搜索策略 — 从贪心到 MCTS

**核心问题**：当前编程代理的搜索策略处于什么水平？未来方向在哪？

| 主题 | 内容 |
|------|------|
| 6.1 搜索策略进化光谱 | 贪心（ReAct）→ 线性回溯（Reflexion）→ 树搜索（ToT）→ MCTS |
| 6.2 当前主流为什么是贪心 | 实际工程约束：token 成本、延迟、状态管理复杂度 |
| 6.3 AlphaGo 深入解剖 | MCTS 四步骤 (Selection/Expansion/Simulation/Backpropagation) 在代码场景的映射 |
| 6.4 代码空间搜索的特殊困难 | 无界状态空间、模糊评价函数、高分支因子 |
| 6.5 前沿进展 | SWE-Search、CodeTree、LATS — 向系统化搜索迈进 |
| 6.6 "编程代理的 AlphaGo 时刻" | 什么条件成熟了这一刻才会到来？ |

**关键论文**：LATS (Zhou 2023), SWE-Search (Antonis 2024), CodeTree (Li 2024)

**实操**：模拟实验 — 同一个 bug fix 任务，分别用贪心策略和树搜索策略，对比路径和效率

## 第三篇：实战篇 — 工程落地（模块 7-8）

### 模块 7：工程实战 — 两大工具深度使用

| 主题 | 内容 |
|------|------|
| 7.1 Claude Code 工程全貌 | 配置体系（CLAUDE.md / settings.json / 项目级 vs 全局）、完整工具链 |
| 7.2 Agent Teams | 多代理协作：团队创建、任务分配、并行执行、进度追踪 |
| 7.3 OpenCode 工程全貌 | 安装配置、多模型接入、LSP 集成、会话管理 |
| 7.4 真实工作流 | 端到端场景：bug fix → 测试 → 提交 → PR，全程代理驱动 |
| 7.5 大型代码库策略 | Monorepo 下的上下文管理、定向探索、分层感知 |
| 7.6 何时不该用代理 | 理解局限性，知道何时手动编码更高效 |

**实操**：
- 用 Claude Code Agent Teams 完成一个多文件功能开发
- 用 OpenCode 配置多模型，对比同一任务的输出

---

### 模块 8：构建你自己的编程代理

| 主题 | 内容 |
|------|------|
| 8.1 从 API 开始 | Claude API / Anthropic SDK 基础：tool_use、streaming、多轮对话 |
| 8.2 最小代理实现 | 100 行代码实现一个能读文件、编辑文件、跑测试的编程代理 |
| 8.3 加入 Grounding | 给你的代理加上代码检索能力（Grep / AST / RAG 任选） |
| 8.4 加入 Feedback | 给你的代理加上测试驱动的自我修正循环 |
| 8.5 加入 Planning | 给你的代理加上 Plan-then-Act 机制 |
| 8.6 企业级考量 | 安全（数据流向）、隐私（本地/自托管模型）、成本控制、审计日志 |

**实操（结业项目）**：从零构建一个领域特定的编程代理，整合课程所有概念

## 课程形式建议

- **3 大篇章，9 个模块**
- 每模块 2-2.5 小时（共约 18-22 小时）
- 每模块结构：机制讲解（30%）+ 论文/算法解读（20%）+ 工程演示（20%）+ 动手实操（30%）
- 结业项目：从零构建一个领域特定的编程代理
