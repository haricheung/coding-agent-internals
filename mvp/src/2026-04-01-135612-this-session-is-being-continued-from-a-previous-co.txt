● 好，现在来回答你关于 model_server.py 是否需要修改的问题。

  你的代码现状分析

  你的 model_server.py 其实已经做了正确的设计：

  1. start_new_session=True（行 841）—— vLLM 子进程在独立的 session group 中启动
  2. Signal handler + os.killpg()（行 951-971）—— 收到 SIGINT/SIGTERM
  时，杀整个进程组

  这意味着：正常退出场景（Ctrl+C）是有保护的。_shutdown 会先 SIGTERM 整个
  pgid，等 5 秒，超时再 SIGKILL。

  为什么这次还是泄漏了？

  你这次是强行杀死了主进程（kill -9 或者类似操作）。问题在于：

  SIGKILL (kill -9)    →  信号不可捕获  →  _shutdown 根本没执行  →  vLLM
  子进程无人管
  SIGINT  (Ctrl+C)     →  信号可捕获    →  _shutdown 执行        →  killpg
  清理子进程
  SIGTERM (kill <pid>)  →  信号可捕获    →  _shutdown 执行        →  killpg
  清理子进程

  所以你的代码在 SIGINT 和 SIGTERM 下没有问题。只有 SIGKILL 
  会导致泄漏，因为内核强制杀进程时不执行任何用户态代码。

  但还有一个更微妙的问题：vLLM 自己的 EngineCore worker。vLLM 内部使用
  multiprocessing 启动 worker 进程，这些 worker 可能：
  - 调用了 start_new_session=True 创建了自己的 session group，脱离了你 killpg
  的范围
  - 或者 vLLM 内部使用 spawn 方式创建进程（而非 fork），子进程不继承 parent 的
  pgid

  这就解释了为什么 os.killpg(os.getpgid(vllm_proc.pid), ...) 可能杀不到所有
  EngineCore worker。

  需要改吗？

  对于课程演示，不需要改。你只需记住：

  - 用 Ctrl+C 退出（触发 _shutdown），别用 kill -9
  - 如果确实要强杀，之后手动清理：ps aux | grep vllm | grep -v grep | awk 
  '{print $2}' | xargs kill

  如果想加固为生产级，可以加两个防线：

  1. atexit 兜底：在非 SIGKILL 的所有退出路径生效

  import atexit
  atexit.register(_shutdown)

  2. 递归杀子进程：用 psutil 遍历进程树，不依赖 pgid

  import psutil
  def _kill_tree(pid):
      try:
          parent = psutil.Process(pid)
          for child in parent.children(recursive=True):
              child.kill()
          parent.kill()
      except psutil.NoSuchProcess:
          pass

  但这对演示来说是过度工程。你现在的代码在正常使用下是没问题的——Ctrl+C
  退出会正确清理所有子进程。你这次遇到的情况是强杀主进程，任何 Python 程序在
  kill -9 下都无法做清理。

  总结：不是你的代码有 bug，是 kill -9 绕过了所有安全网。以后用 Ctrl+C 或 kill 
  <pid>（不带 -9）退出就行。
